# PCA Color Matching Notes

## Problem

The 2D and 3D PCA capsules can receive the same default `color_values` palette but still assign different colors to the same sample groups. The palette itself is not the root problem. The mismatch happens because group-to-color assignment is handled differently by the plotting backends.

- `plot_pca_2d()` uses `ggplot2::scale_colour_manual(values = color_values)`.
- `plot_pca_3d()` passes `colors = color_values` to `plotly::plot_ly()`.
- When `color_values` is unnamed, ggplot and plotly may assign the same palette to group labels using different category ordering or trace handling.

The expected default behavior is that every PCA plot assigns the same group labels to the same default colors. For example, with groups `A`, `B`, and `C`, all PCA outputs should consistently map:

```text
A -> #5954d6
B -> #e1562c
C -> #b80058
```

This should apply to:

- standalone 2D PCA capsule output
- standalone 3D PCA capsule output
- PCA output generated by the filter capsule/package flow
- PCA output generated by the normalization capsule/package flow

## Package-Level Change Needed

Fix this in the MOSuite package instead of in individual capsule wrappers.

Add one shared helper that normalizes PCA colors before plotting. The helper should:

1. Accept `color_values` and the observed group labels.
2. Preserve existing names when `color_values` is already a named vector for those labels.
3. When `color_values` is unnamed, assign names deterministically using sorted unique group labels.
4. Return the original vector unchanged if there are more groups than colors, so the current failure mode is not hidden.

Likely helper shape:

```r
match_group_colors <- function(color_values, group_values) {
  group_values <- sort(unique(as.character(group_values)))
  if (is.null(names(color_values)) || !all(group_values %in% names(color_values))) {
    if (length(group_values) <= length(color_values)) {
      color_values <- stats::setNames(color_values[seq_along(group_values)], group_values)
    }
  }
  color_values
}
```

Likely location: [R/colors.R](R/colors.R) or [R/plot_pca.R](R/plot_pca.R). Prefer [R/colors.R](R/colors.R) if the helper may also be useful outside PCA.

## Files To Inspect Or Change

- [R/plot_pca.R](R/plot_pca.R)
  - Apply the shared color matching before `ggplot2::scale_colour_manual()` in `plot_pca_2d()`.
  - Apply the same shared color matching before `plotly::plot_ly(colors = ...)` in `plot_pca_3d()`.
  - This is the core fix for standalone 2D and 3D PCA consistency.

- [R/filter.R](R/filter.R)
  - Filter-generated PCA passes `colors_for_plots` into `plot_pca()`.
  - Avoid stripping names from `moo@analyses[["colors"]][[group_colname]]` before PCA plotting. The current use of `as.vector(colors_for_plots)` can remove group names and recreate the mismatch.

- [R/normalize.R](R/normalize.R)
  - Normalization-generated PCA also passes `colors_for_plots` into `plot_pca()`.
  - Preserve or normalize names so normalization PCA output uses the same group mapping as standalone PCA.

- [R/batch-correction.R](R/batch-correction.R)
  - Not part of the immediate capsule request, but it also creates PCA plots with `colors_for_plots`. Check it for the same pattern while fixing all PCA outputs.

## Tests To Add

Add tests in [tests/testthat/test-plot_pca.R](tests/testthat/test-plot_pca.R) or a nearby color-specific test file.

Recommended test coverage:

1. `plot_pca_2d()` with unnamed default colors maps groups `A`, `B`, `C` to the first three default colors.
2. `plot_pca_3d()` with unnamed default colors maps groups `A`, `B`, `C` to the same first three default colors.
3. Named colors are preserved exactly when passed in.
4. Filter-generated PCA output preserves group color names.
5. Normalization-generated PCA output preserves group color names.

The important assertion is not that the palette strings match. The assertion must check the group-to-color mapping.

## Capsule Follow-Up

A temporary wrapper-level fix was made in the 3D PCA capsule while diagnosing the mismatch. Once the package-level fix lands and the capsule uses the updated MOSuite package, remove that wrapper workaround so color matching is owned by MOSuite itself.
